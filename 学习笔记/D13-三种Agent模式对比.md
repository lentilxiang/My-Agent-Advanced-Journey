# D13：三种 Agent 模式对比 + 选型决策框架

> 目标：将 ReAct、Plan-and-Execute、Reflection 放在一张图上对比，建立"看到任务就知道该用哪种模式"的判断力。

---

## 1. 一张图看懂三种模式

```
                        Agent 设计模式全景

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │   ReAct                    Plan-and-Execute        Reflection   │
  │   边走边想                  先规划后执行              做→查→改    │
  │                                                                 │
  │  Thought→Action→Obs        Plan→Exec→Replan        Gen→Crit→Rev │
  │      ↑         │               ↑                    ↑     │    │
  │      └─────────┘               │                    └─────┘    │
  │      横向探索                  纵向+横向              纵向打磨    │
  │      (广度优先)                (先深度后广度)         (深度优先)  │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  任务类型 → 选哪种？

  探索性任务 ────────────────→ ReAct
  (不确定几步，需要边搜边决定)

  结构性任务 ────────────────→ Plan-and-Execute
  (可预先分解，有明确步骤)

  质量要求高的任务 ──────────→ Reflection
  (有评价标准，可以多轮打磨)

  复杂任务 ──────────────────→ 组合
  (Plan-and-Execute 管方向 + ReAct 管执行 + Reflection 管质量)
```

## 2. 详细对比

### 2.1 架构对比

```
              ReAct               Plan-and-Execute        Reflection
  ────────    ─────────────────    ────────────────────    ───────────────────
  组件数       1个循环              3个组件                 3个组件
              (单一循环)           (Planner/Exec/Replan)   (Generator/Critic/Reviser)

  决策粒度     下一步调哪个工具      下一步做什么步骤         这个输出好不好

  信息流       线性追加              分层传递                循环迭代
              messages 不断增长     Plan → 各Step → 汇总     同一输出多轮加工

  LLM 角色     推理 + 行动           规划 + 执行 + 调整       生成 + 评价 + 修改

  输出特点     逐步积累              分步产出                 单份打磨
```

### 2.2 性能对比

```
              ReAct          Plan-and-Execute      Reflection
  ────────    ────────────   ─────────────────     ──────────────
  LLM 调用量   中（N轮≈N次）   高（Plan 1次 + 每步1-3次） 高（每轮2次：Critic+Revise）
  Token 消耗   中             高（长prompt+长结果）   高（重复传同一份输出）
  延迟        中             较高（串行步骤）        高（多轮迭代）
  首次响应     快（第1轮就有） 慢（需等全部步骤完成）  快（第1轮就有初稿）
  最终质量     中（无检查机制） 中高（结构有保证）     高（Critic把关）
  可靠性       中（可能跑偏）  中高（Plan给方向）     高（多轮改进+退化检测）
```

### 2.3 失败模式

```
ReAct 的失败模式：
  ① 无限循环：反复调同一工具得不到有用结果
  ② 跑偏：没有全局规划，第5步已经偏离了原始问题
  ③ 上下文爆炸：每轮追加 Thought+Action+Obs，token 疯涨
  ④ 信息不足就下结论：没搜到就基于幻觉编造

Plan-and-Execute 的失败模式：
  ① 初始计划错误：Planner 分解不合理 → 全盘皆输
  ② 步骤间耦合：Step 2 依赖 Step 1 的结果但 Step 1 产出不对
  ③ 过度分解：把简单任务拆成 10 步 → LLM 调用量爆炸
  ④ Replanner 失效：计划需要调整但没触发生效

Reflection 的失败模式：
  ① 过度修改（越改越差）：Critic 意见偏差 → Reviser 改错方向
  ② 循环停滞：来回改同样的东西 → 输出在两端反复
  ③ Critic 误判：该 PASS 的打了 FAIL，该 FAIL 的放了 PASS
  ④ 成本失控：每轮 2 次 LLM 调用（Critic + Reviser），3 轮 = 7 次调用
```

## 3. 选型决策框架

### 3.1 决策树

```
收到一个任务 →
  │
  ├─ 能一步完成？（如 "1+1=?"）
  │   └─ YES → 纯 LLM 回答（不需要 Agent）
  │
  ├─ 需要外部信息且不确定几步？
  │   └─ YES → ReAct
  │       例: "A和B谁更早创业？"、"北京现在几点？"
  │
  ├─ 任务可以预先分解成分步执行？
  │   └─ YES → Plan-and-Execute
  │       例: "写一份调研报告"、"帮我订机票+酒店+行程"
  │       注意: 如果步骤间耦合强 → 每步内部用 ReAct 执行
  │
  ├─ 输出有明确质量标准，需要打磨？
  │   └─ YES → Reflection
  │       例: "写生产级代码"、"写正式邮件"、"数学证明"
  │       注意: Reflection 可以和前两种叠加
  │
  └─ 复杂任务（以上多项叠加）？
      └─ YES → 组合模式
          Plan-and-Execute 做顶层 → ReAct 做每步 → Reflection 做终审
```

### 3.2 按任务类型推荐

| 任务类型 | 推荐模式 | 原因 |
|---------|---------|------|
| 事实查询（"XX是什么"） | 纯 LLM 或 单次搜索 | 不需要多步 |
| 多步搜索+推理 | **ReAct** | 边搜边推理，灵活 |
| 数据收集+汇总 | **ReAct** | 逐步收集，动态调整搜索策略 |
| 报告/文章撰写 | **Plan-and-Execute** | 大纲→分节→汇总 |
| 多步骤操作（订票+酒店） | **Plan-and-Execute** | 步骤明确，可预先排好 |
| 代码生成（需高质量） | **Reflection** | 生成→Review→改 |
| 数学证明 | **Reflection** | 写→自查→修正 |
| 复杂研究任务 | **Plan→ReAct→Reflect** | 三者组合 |
| 客服对话 | **ReAct** | 多轮互动，每轮决策 |
| 代码审查 | **Reflection** | 读代码→分析→建议 |

## 4. 生产级组合模式

```
实际应用中最常见的组合：

  ┌─────────────────────────────────────────────┐
  │          Plan-and-Execute (顶层编排)          │
  │                                             │
  │  Step 1: 调研 XX 领域                        │
  │    └─ ReAct 循环（搜→读→搜→总结）             │
  │  Step 2: 分析 XX 趋势                        │
  │    └─ ReAct 循环（搜→计算→搜→总结）           │
  │  Step 3: 撰写报告                            │
  │    └─ Reflection 循环（写→审→改→审→输出）     │
  │                                             │
  └─────────────────────────────────────────────┘

  生产建议（来自 Anthropic Building Effective Agents）：
    "从最简单的方案开始，只在必要时增加复杂度。
     不是每个任务都需要 Agent。
     如果可以用单次 LLM 调用解决，就不要用 Agent 循环。"
```

```text
[ ] 练习 1：同一个任务，分别用三种模式跑一遍
           "帮我研究AI Agent领域的最新进展并写一份简报"

           统计决策准确率

[ ] 练习 3：组合模式实战
           用 Plan-and-Execute 做顶层（3个步骤）
           每个步骤内部用 ReAct 执行
           最后一步用 Reflection 打磨
```

## 6. 关键收获

```
三种模式各司其职：
  ReAct = 探索者 → 不知道需要什么信息时用它
  Plan-and-Execute = 建筑师 → 知道要干什么时先画图
  Reflection = 质检员 → 产出有了，打磨到满意

一句话选型：
  "不确定几步 → ReAct"
  "能分解步骤 → Plan-and-Execute"
  "要高质量 → Reflection"
  "都想用 → 组合"

第 2 周至此完结。下周进入多 Agent 协作 + 记忆系统。
```
