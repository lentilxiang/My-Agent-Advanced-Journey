# D10：工具系统设计

> 目标：设计一套可复用的工具系统——注册、描述、校验、选择、容错。这是 Agent 工程的骨架。

---

## 1. 为什么要设计工具系统？

D04/D06/D09 中，工具是"写死的"——每个工具手写一个类，prompt 手写，参数自己解析。这在生产环境中无法维护。

```
D04/D06/D09 的做法：
  tools = [SearchTool(), CalculatorTool(), DateTimeTool()]
  → 工具少（3-5个），手写就够了
  → 参数校验靠工具内部 try/except
  → 工具描述手写，质量因人而异

生产环境的需求（10+ 工具，多人协作）：
  - 新增一个工具 = 写一个函数 + 一行注册 → 不应该改其他代码
  - 工具描述自动生成 → 避免人工写错/写漏
  - 参数在调用前校验 → 不要等执行失败才知道参数错了
  - 工具太多 LLM 选不对 → 需要筛选/路由策略
  - 工具调用失败自动重试 → 区分瞬时错误 vs 参数错误
```

**D10 要做的事：把"写工具"从手艺活变成工程化。**

## 2. 工具注册机制

### 2.1 三种注册方式

```
方式 A：手动列表（D04/D06/D09）
  tools = [ToolA(), ToolB(), ToolC()]
  适用：工具 < 5 个，原型阶段

方式 B：装饰器注册（D10 推荐）
  @tool_registry.register
  def search(query: str) -> str:
      """在互联网上搜索信息。"""
      ...

  优点：新增工具只需写函数 + 加装饰器，不改其他代码
        注册表自动维护，不需要手动维护列表

方式 C：自动发现
  扫描 tools/ 目录 → 自动加载所有工具
  适用：大型项目，工具可能来自不同团队
```

### 2.2 注册表设计

```python
class ToolRegistry:
    """工具注册中心"""
    _tools: dict[str, Tool] = {}

    def register(self, func=None, *, name=None, description=None):
        """装饰器方式注册工具"""
        def decorator(fn):
            tool_name = name or fn.__name__
            desc = description or fn.__doc__ or ""
            # 从 type hints 自动生成参数 schema
            schema = type_hints_to_json_schema(fn)
            self._tools[tool_name] = Tool(
                name=tool_name,
                description=desc.strip(),
                parameters=schema,
                executor=fn,
            )
            return fn
        return decorator(func) if func else decorator

    def get_tool(self, name: str) -> Tool: ...
    def list_tools(self) -> list[Tool]: ...
    def to_openai_format(self) -> list[dict]: ...  # → OpenAI tools 参数
    def to_anthropic_format(self) -> list[dict]: ... # → Anthropic tools 参数
```

## 3. 工具描述模板

### 3.1 好描述 vs 差描述

工具描述是 LLM 选择工具的**唯一依据**。描述差 → 选错工具 → 整个 Agent 跑偏。

```
差描述（D04 风格）：
  {"name": "search", "description": "搜索东西"}
  → LLM 不知道什么时候该用、返回什么、有什么限制

好描述（生产标准）：
  {
    "name": "web_search",
    "description": (
      "在互联网上搜索实时信息。"
      "适用场景：需要最新数据（新闻、价格、天气）、验证事实。"
      "不适用：已知常识、数学计算、代码生成。"
      "返回：匹配的搜索结果摘要列表（最多 10 条）"
    ),
    "parameters": {
      "query": {
        "type": "string",
        "description": "搜索查询词。应具体且信息丰富，"
                       "如 '2024年诺贝尔物理学奖获得者' 而非 '诺贝尔奖'。"
      }
    }
  }
```

### 3.2 描述模板六要素

```
① 功能说明    → "在互联网上搜索实时信息"
② 适用场景    → "适用于需要最新数据时，如新闻、股价、天气"
③ 不适用场景  → "不适用于已知常识、数学计算、代码生成"（帮 LLM 排除）
④ 返回格式    → "返回搜索结果摘要列表，每条包含标题和链接"
⑤ 限制条件    → "每次搜索最多返回 10 条结果"
⑥ 参数说明    → 每个参数：类型 + 含义 + 示例值 + 约束（enum/min/max）
```

## 4. 参数校验

### 4.1 为什么要在调用前校验？

```
LLM 生成的参数不一定合法：
  - get_weather(city="")        ← 空字符串
  - calculator(expr="1/0")      ← 除零
  - send_email(to="notanemail") ← 格式错误

两难：
  校验太松 → 工具执行出错，浪费一轮对话
  校验太严 → 频繁拒绝，模型反复重试

最佳实践：
  - 类型 + 格式：严格校验（空值、类型不匹配 → 直接拒绝，告诉模型修正）
  - 业务逻辑：宽松校验（1/0 让工具自己报错，模型从 Observation 中学习）
```

### 4.2 Pydantic 校验

```python
from pydantic import BaseModel, Field

class WebSearchInput(BaseModel):
    query: str = Field(..., min_length=1, max_length=500,
                       description="搜索查询词")

class CalculatorInput(BaseModel):
    expression: str = Field(..., min_length=1, max_length=200,
                            description="数学表达式，如 '32 - 28'")

# 校验
try:
    params = WebSearchInput(query=llm_generated_query)
except ValidationError as e:
    return ToolError(f"参数错误: {e.errors()}", retryable=True)
```

### 4.3 type hints → JSON Schema 自动生成

```python
import inspect
from typing import get_type_hints

def function_to_json_schema(func) -> dict:
    """从函数的 type hints + docstring 自动生成工具 schema"""
    hints = get_type_hints(func)
    sig = inspect.signature(func)
    doc = func.__doc__ or ""

    properties = {}
    required = []

    for name, param in sig.parameters.items():
        if name in ("self", "cls"):
            continue
        py_type = hints.get(name, str)
        json_type = python_type_to_json_type(py_type)
        properties[name] = {
            "type": json_type,
            "description": extract_param_description(doc, name) or f"{name} 参数",
        }
        if param.default is inspect.Parameter.empty:
            required.append(name)

    return {
        "name": func.__name__,
        "description": extract_func_description(doc),
        "parameters": {
            "type": "object",
            "properties": properties,
            "required": required,
        }
    }
```

## 5. 多工具选择策略

### 5.1 问题：工具太多怎么办？

```
10 个工具 → LLM 选择准确率 ~90%
20 个工具 → LLM 选择准确率 ~75%
50 个工具 → LLM 选择准确率 ~50%（开始乱选）
```

### 5.2 四种策略

```
策略 A：全量传入（工具 ≤ 10 个）
  直接把所有工具传给 LLM
  最灵活，但工具太多时准确率下降

策略 B：分类路由（工具 10-50 个）
  先让 LLM 判断任务类型 → 再传入该类型的工具子集
  
  例如：
    用户问天气 → 路由到 "信息查询" 类 → 传入 search/weather/news 工具
    用户问计算 → 路由到 "数据处理" 类 → 传入 calculator/unit_converter/stats 工具

策略 C：语义检索（工具 50+ 个）
  用 embedding 对工具描述做向量化
  根据用户 query 检索最相关的 top-5 工具
  只传这 5 个给 LLM

策略 D：混合策略（生产推荐）
  ① embedding 检索 top-10
  ② LLM 分类筛选 top-5
  ③ 传入 LLM 做最终选择
```

## 6. 错误处理与重试

### 6.1 错误分类

```
参数错误（不可重试）：
  - 类型不匹配、缺少必填字段、格式错误
  → 返回明确错误信息，让 LLM 重新生成参数

业务错误（不可重试）：
  - 用户不存在、权限不足、数据不满足条件
  → 返回业务语义描述，LLM 可能切换策略

瞬时错误（可重试）：
  - 网络超时、限流 429、服务繁忙 503
  → 指数退避重试：1s → 2s → 4s（最多 3 次）

系统错误（降级）：
  - 服务不可用、数据库挂了
  → 告知用户稍后重试，给出替代方案
```

### 6.2 重试装饰器

```python
class ToolError(Exception):
    def __init__(self, message: str, retryable: bool = False):
        self.message = message
        self.retryable = retryable

def with_retry(max_retries: int = 3, base_delay: float = 1.0):
    """工具调用的重试装饰器"""
    def decorator(func):
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except ToolError as e:
                    if not e.retryable:
                        raise  # 不可重试，直接抛
                    last_error = e
                    time.sleep(base_delay * (2 ** attempt))
            raise ToolError(f"重试 {max_retries} 次后仍失败: {last_error}")
        return wrapper
    return decorator
```

## 7. 安全性

```
① 工具白名单：用户会话级别，根据场景只暴露必要工具
② 参数校验：执行前严格校验，拒绝越权操作
③ 人工确认：破坏性操作（删除/转账）返回确认请求，不直接执行
④ 输入过滤：用户输入中的明显注入模式（"忽略指令"、"调用所有工具"）做标记
⑤ 返回脱敏：工具返回的敏感信息（手机号/邮箱）脱敏后再注入 LLM 上下文
```
