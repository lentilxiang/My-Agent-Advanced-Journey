# 第 2 周 面试题精选：Agent 设计模式

> 覆盖 D08-D14 全部知识点，题目来自阿里巴巴、美团、华为、字节、腾讯、微软、百度、蚂蚁等公司真实面经。

---

## 一、ReAct 模式（D08/D09）

### Q1: ReAct 框架的工作原理？Thought/Action/Observation 分别是什么？

**真题来源：** 阿里巴巴、美团（二面）、华为

**参考答案：**

ReAct = Reasoning（推理）+ Acting（行动），交替进行思考和行动。

三要素：
- **Thought**：LLM 用自然语言分析当前状态，规划下一步该做什么。不是代码注释，是模型"自言自语"的推理过程。
- **Action**：决定调用哪个工具、传什么参数。在 FC 模式下是结构化 `tool_calls` JSON，在文本协议下是 `Action: tool_name`。
- **Observation**：工具执行后的返回结果——Agent "看到的"环境反馈。用于更新认知、影响下一步决策。

循环流程：
```
Thought: 需要查 XX 信息
  → Action: search("XX")
    → Observation: "搜索结果: XX 是..."
      → Thought: 拿到数据了，还需要 YY
        → Action: search("YY")
          → Observation: "YY 是..."
            → Thought: 信息够了
              → Action: Finish → 输出答案
```

**延伸追问：ReAct 和 CoT（Chain-of-Thought）有什么区别？**
- CoT = 纯推理，不调工具，容易产生幻觉（凭记忆编造）
- ReAct = 推理 + 行动，通过工具获取真实信息，有 Observation 锚定事实
- 数据：ReAct 在 HotpotQA 上比 CoT 提升约 6%，在 FEVER 上提升约 10%

### Q2: Function Calling 和 ReAct 是什么关系？互斥还是互补？

**真题来源：** 字节跳动、2025 高频新题

**参考答案：**

互补关系，解决不同层次的问题：
- **Function Calling** → "能力层"：让 LLM 输出结构化的工具调用指令 `{name: "search", arguments: {...}}`，解决"怎么调用工具"。
- **ReAct** → "策略层"：定义 Agent 的决策循环 Thought → Action → Observation，解决"什么时候调、调哪个、结果怎么用"。

类比：ReAct 是大脑（做决策），Function Calling 是手（执行操作）。

工程实现中它们总是一起出现——LangChain AgentExecutor 用 ReAct 做决策循环，底层用 OpenAI FC API 执行工具调用。

---

### Q3: Function Calling 的底层实现原理？模型是如何学会"何时调用工具、如何生成参数"的？

**真题来源：** OpenAI、百度、蚂蚁

**参考答案：**

分三个层面：

**训练层面（模型怎么学会的）：**
- SFT 阶段：在训练数据中加入大量工具调用样本。输入是 system prompt 含 tools 定义 + 用户问题，期望输出是 assistant 返回结构化的 `tool_calls`。
- 模型学到的不是"执行函数"，而是输出格式："当我看到 tools 定义且用户问题需要调用工具时，我应该输出 tool_calls 字段而不是普通文本。"
- 这个能力写在了模型权重里。

**推理层面（模型怎么决定调哪个）：**
- 模型前向传播时，根据 tools 描述和用户 query 做语义匹配
- 内部计算后直接输出 `tool_calls` token（name + JSON arguments）
- 这个推理过程对外不可见（不像显式 ReAct 有 Thought 文本）

**执行层面（谁真正执行了函数）：**
- 模型不执行任何函数！只输出调用指令
- 开发者代码读取 `response.tool_calls` → 执行对应函数 → 结果返回模型

---

### Q4: Function Calling 和 JSON Mode 有什么区别？

**真题来源：** Function Calling 专题（通用高频）

**参考答案：**

| | Function Calling | JSON Mode |
|---|---|---|
| 目的 | 模型输出工具调用意图 | 模型输出符合 schema 的 JSON |
| 输出 | `tool_calls` 字段（name + arguments） | `content` 字段（纯 JSON 文本） |
| 模型训练 | 需要专门 FC 训练 | 不需要，JSON 输出是通用能力 |
| 执行 | API 自动解析 tool_calls | 开发者 `json.loads()` 解析 |
| 适用 | 工具调用场景 | 结构化数据提取 |

## 二、工具系统设计（D10）

### Q5: 如何设计一个好的工具描述？什么样的功能应该做成 Tool？

**真题来源：** 牛客高频拷打题、微软

**参考答案：**

**什么该做成 Tool：** LLM 本身做不好的事情——实时数据查询、精确计算、外部系统操作、文件读写、代码执行。

**好描述的六要素：**
1. **名称**：动词+名词，清晰表达意图（`create_issue` 而非 `issue`）
2. **功能说明**：一句话说明做什么
3. **适用场景**：什么时候该用这个工具
4. **不适用场景**：什么时候不该用（帮 LLM 排除错误选项）
5. **返回格式**：返回值是什么样子
6. **参数说明**：每个参数的类型、含义、示例值、约束（enum/min/max）

**对比示例：**
```json
// ❌ 差
{"name": "search", "description": "搜索东西"}

// ✅ 好
{"name": "web_search",
 "description": "在互联网上搜索实时信息。适用于需要最新数据时（新闻、价格、天气）。不适用于已知常识、数学计算。返回搜索结果摘要。",
 "parameters": {
   "query": {"type": "string", "description": "搜索查询词，应具体且信息丰富。如'2024诺贝尔奖'而非'诺贝尔奖'"}
 }}
```

### Q6: 多工具场景下，如何避免模型选错工具？

**真题来源：** Function Calling 专题、字节跳动

**参考答案：**

**预防层：**
- 工具描述清晰，包含适用/不适用场景
- 工具数量控制在 10-20 个以内
- 用分类路由减少候选工具集
- Few-shot 示例教模型正确的选择模式

**检测层：**
- 工具返回值验证（类型检查、合理性检查）
- 参数校验（类型、范围、格式）

**恢复层：**
- 参数错误 → 返回明确错误信息，让 LLM 修正重试
- 选错工具 → LLM 从 Observation 中发现自己选错了，下一轮自动纠正

**高级策略：**
- 工具数量 > 20 → 分类路由（先判断任务类型，再传该类型的工具子集）
- 工具数量 > 50 → 语义检索（embedding 检索 top-5 相关工具）

---

### Q7: Agent 调用工具不正确怎么办？

**真题来源：** 字节跳动

**参考答案：**

四层防护体系：
1. **预防**：优化工具描述、Few-shot 示例、参数校验（类型/范围/格式）
2. **检测**：工具返回值验证、类型检查、超时检测
3. **恢复**：重试机制（指数退避）、fallback 工具、人工介入
4. **学习**：记录错误案例，用于优化 prompt 或微调模型

## 三、Agent 循环设计（D08/D09/D10）

### Q8: Agent Loop 的核心设计考量有哪些？

**真题来源：** 百度（社招二面）、2025 高频新题

**参考答案：**

**① 终止条件（必须有明确的退出机制）：**
- max_steps：硬限制最大步数
- 目标达成判断：LLM 自主输出 Finish
- 用户中断：支持手动停止

**② 上下文管理（防止 token 爆炸）：**
- 滑动窗口：保留最近 K 轮完整内容
- 摘要压缩：更早内容用 LLM 摘要
- 策略：系统提示 + 最近 K 轮原文 + 历史摘要

**③ 错误处理（每层兜底）：**
- 工具层：重试（指数退避）+ 回退（备用工具）
- 推理层：循环检测 + 死胡同检测
- 任务层：总超时 + Token 预算上限

**④ 状态管理：**
- 工作记忆：当前任务的中间变量
- 避免重复工作：已获取的信息不要重复搜索

**⑤ 可观测性：**
- 每步记录 Thought/Action/Observation
- 追踪 Token 消耗和延迟
- 支持单步回放调试

---

### Q9: Agent 如何检测和防止无限循环？

**真题来源：** 通用八股文、腾讯

**参考答案：**

**检测方法：**
- 重复检测：最近 N 步（N=3~5）调用的工具和参数完全相同 → 陷入循环
- 语义重复检测：最近 N 步的 Thought 内容高度相似 → 在绕圈
- 无进展检测：连续 N 步没有获得新的有效信息

**防护措施：**
- max_turns 硬限制
- 检测到循环 → 强制切换策略（换工具、换搜索词）
- 切换策略仍无效 → 告知用户无法完成，请求人工介入

## 四、Plan-and-Execute 模式（D11）

### Q10: Agent 怎么做任务规划？ReAct 还是 Plan-Execute？

**真题来源：** 牛客高频拷打题、百度

**参考答案：**

**ReAct（边走边想）：**
- 适用：探索性任务，不确定需要几步
- 优点：灵活，能应对意外情况
- 缺点：可能走弯路，没有全局视野

**Plan-and-Execute（先规划后执行）：**
- 适用：结构化任务，可以预先分解
- 优点：LLM 调用次数少，进度可见，支持人工审批
- 缺点：初始计划可能不准，遇到变化不灵活

**生产实践——混合方案（Adaptive Planning）：**
1. 先粗粒度 Plan（顶层分几步）
2. 每步内部用 ReAct 动态执行
3. 遇到异常时 Replan（调整后续计划）

一句话：Plan 给方向，ReAct 给灵活。两者组合才是生产级方案。

## 五、Reflection 模式（D12）

### Q11: Reflection 模式是如何工作的？

**真题来源：** 通用知识

**参考答案：**

Reflection = Generator（生成）+ Critic（审查）+ Reviser（修改）循环。

流程：
1. Generator 生成初始输出
2. Critic 从正确性、完整性、清晰度、安全性等维度评估
3. 如果通过 → 输出；如果不通过 → 生成改进建议
4. Reviser 根据反馈针对性修改，保留 Critic 认可的部分
5. 回到步骤 2，直到通过或达到最大轮次

**防止过度修改：**
- 保留每个版本 → 最终选 Critic 评分最高的
- 每次只改被指出的具体问题
- 限制修改轮次（2-3 轮）
- 检测退化 → 新版本不如旧版本时回退

## 六、错误处理（D10 核心）

### Q12: 工具调用出错了怎么处理？错误分类和重试策略？

**真题来源：** 腾讯、微软、小红书、字节跳动

**参考答案：**

**四级错误分类：**

| 错误类型 | 示例 | 处理策略 |
|---------|------|---------|
| **参数错误** | 缺少必填参数、类型不匹配 | 返回明确错误信息，让模型修正参数重试 |
| **业务错误** | 用户不存在、权限不足 | 返回业务语义描述，模型换策略 |
| **瞬时错误** | 网络超时、限流 429 | 指数退避重试：1s → 2s → 4s（最多 3 次） |
| **系统错误** | 服务不可用 500 | 返回友好提示，告知用户稍后重试 |

**微软 RARF 框架：**
- **R**ecognize：识别错误类型
- **A**dapt：自适应策略（指数退避重试）
- **R**oute：路由替代（切换到 fallback 工具）
- **F**eedback：反馈用户（让错误处理成为 Agent 的能力）

**关键原则：不要吞掉错误。让模型从错误中学习，在下一次尝试中修正。**

---

### Q13: 如何设计 Agent 的错误恢复和容错机制？

**真题来源：** 微软、腾讯、通用八股文

**参考答案：**

需要在四个层次设计容错：

- **工具层**：重试逻辑（指数退避，最多 3 次）、备用工具回退（搜索引擎 A 失败 → 切 B）、超时控制（15-30s）
- **推理层**：循环检测（最近 N 步是否重复同一行动）、死胡同检测（连续多步无进展）、上下文溢出保护
- **任务层**：总步骤上限（max_steps）、总 token 预算、总时间超时
- **输出层**：结构化输出解析失败时重试、Guardrails 检查输出合规性

兜底策略：所有自动处理失败 → 优雅降级，向用户说明情况并请求帮助，而非返回幻觉答案。

## 七、三种模式对比（D13）

### Q14: Agent 设计模式有哪几种？各自的适用场景是什么？

**真题来源：** 百度（二面）、通用知识

**参考答案：**

三种核心模式 + 选型口诀：

| 模式 | 一句话 | 适用场景 | 不适用 |
|------|--------|---------|--------|
| **ReAct** | 边走边想 | 探索性任务、不确定几步 | 需要全局规划的任务 |
| **Plan-and-Execute** | 先画地图再走 | 结构性任务、可预先分解 | 高度不确定的探索 |
| **Reflection** | 做→查→改 | 有质量标准的任务 | 实时性要求高的场景 |

选型口诀：
- 不确定几步 → ReAct
- 能分解步骤 → Plan-and-Execute
- 要高质量 → Reflection
- 复杂任务 → 三者组合（Plan 管方向 + ReAct 管执行 + Reflection 管质量）

---

### Q15: 单 Agent vs 多 Agent 系统，如何选择？

**真题来源：** 通用知识

**参考答案：**

- **单 Agent 适合：** 任务步骤线性可控、不需要多领域专业知识、对延迟敏感、系统简单性优先
- **多 Agent 适合：** 任务涉及多个专业领域、子任务可并行执行、需要交叉检验提高准确性、单个上下文窗口装不下

实践建议：**先用单 Agent 解决，确认瓶颈后再拆分为多 Agent。** 多 Agent = 多倍成本 + 调试复杂度急剧上升。

## 八、高频速查

| 面试题 | 一句话答案 |
|--------|-----------|
| ReAct 是什么？ | 推理+行动交替循环：Thought→Action→Observation |
| ReAct vs CoT？ | CoT 纯推理（可能幻觉），ReAct 推理+工具（真实数据锚定） |
| ReAct vs FC？ | ReAct 是策略层（决策循环），FC 是能力层（调用格式），互补 |
| FC 模型怎么学会的？ | SFT 训练数据包含 tool_calls 样本，模型学会输出结构化调用指令 |
| 好工具描述六要素？ | 名称、功能、适用/不适用场景、返回格式、参数约束 |
| 工具太多怎么选？ | <20 全量传入，20-50 分类路由，>50 语义检索 |
| 错误怎么分类？ | 参数错/业务错/瞬时错/系统错，只重试瞬时错误 |
| Plan vs ReAct？ | Plan 给方向，ReAct 给灵活，生产用混合方案 |
| Reflection 怎么防过度修改？ | 保留版本+限制轮次+检测退化+只改被指出的问题 |
| Agent Loop 三大考量？ | 终止条件、上下文管理、错误处理 |
